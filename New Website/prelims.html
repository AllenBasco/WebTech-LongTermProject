MESSAGE HEADERS

  1. GET
    • Transfer a current selected representation of the resource identified by the request URI
    • Fetch resources
  2. HEAD
    • Same as GET with one difference (returns only the statusline and the headers)
    • Used to retrieve metadata about the entity implied by the request without transferring the entity itself
  3. PUT
    • Store the enclosed entity in the message body under the specified request URI
  4. OPTIONS
    • Tell you what you can do or request information about communication
  5. TRACE
    • Used to request a loop▹back of the request message
    • Used for testing/diagnostics of the request/response chain
  6. CONNECT
    • Use only for HTTPS communication
    • Request the establishment of a tunel to the destination origin server, and if successful thereafter restrict its behavior to blind forwarding of packets
  1. Safe Method
    • A method that doesn't make any change.
  2. Idempotent Method
    • A method that you can use a method multiple times but get the same result.
  3. Cacheable method
    • The response generated could be cachd or not.
  1. Cache Control
    • allow us to control caching and the information won't be saved
  2. Connection
    • used to establish persistent connection
  3. Date
    • to control the lifetime of cache resources
  4. Pragma
    • general directive in older version of HTTP
  5. Trailer
    • header that is pass/provided after the data
  6. Transfer
    • to notify the server it expects to receive additional fields in the trailer
  7. Upgrade
    • to switch to a different protocol
  8. Via
    • where the message was passed
  9. Warning
    • general warning about possible problems with the entity body
  1. ACCEPT
    • A field which has content that are acceptable for response.
  2. ACCEPT-CHARSET
    • Character sets that are viable for the response.
  3. ACCEPT-ENCODING
    • These are the encodings that are acceptable for the response.
  4. ACCEPT-LANGUAGE
    • These are the languages that are suitable for the response.
  5. AUTHORIZATION
    • This needs your credentials for aunthentication so as to proceed.
  6. EXPECT
    • It needs certain behaviors that are demanded by the client.
  7. FROM
    • It contains the users email address.
  8. HOST
    • It contains the domain name of the server and also the port number which has the Transmission Control Protocol port.
  9. IF-MATCH
    • This request header makes a conditional request in which the client matches with entity located on the server.
  10. IF-MODIFIED
    • It makes a method conditional. It is noted that 304 will be encountered.
  11. IF-NONE-MATCH-IF-RANGE
    • The client has sources that were obtained from previous resources.
  12. IF-UNMODIFIED-SINCE
  13. MAX-FORWARDS
    • It sends the response only if the entity has not been modified since a specified time.
  14. PROXY-AUTHORIZATION
    • Credentials are needed to access this proxy.
  15. RANGE
    • It has a partial copy but it has missing features. It requests a copy that is complete and suitable.
  16. REFERER
    • It is an optional header field. It allows links which are not suitable to be maintainted. It allows the server to generalete list of back-links such as references.
  17. TE
    • Also known as "Transfer Encodings", It indicates which transformation is needed to be applied on the message body to bes ent to the sender and client.
  18. USER-AGENT
    • It contains the user agent that gave the request.
